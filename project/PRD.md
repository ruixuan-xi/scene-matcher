# 场景收集与智能去重沉淀智能体 — PRD

> 版本：V2.2 | 更新：2026-07-02
>
> **V2.2 重大变更**：从 `psycopg2` 直连数据库改为 **Supabase REST API** 模式。Supabase DB 端点仅 IPv6 可用，沙箱网络无法路由，故所有数据读写走 REST API。向量相似度在本地用 numpy 批量计算。

---

## 1. 项目概述

### 1.1 目标
帮助企业收集各部门的工作场景需求，自动判断是否与已有场景重复，避免重复入库，沉淀高价值新场景。

### 1.2 核心流程
```
用户上传 Excel
  → 模板校验与智能映射（LLM 映射表头）
      ├─ 列名完全匹配 → 进入下一步
      ├─ 列名可智能映射 → 生成 {文件名}_待补充.xlsx，退回补充
      └─ 严重缺列 → 直接退回，要求按模板重新上传
  → 步骤详细度检查
      ├─ 无分步标记 → 检查总长≥30字
      └─ 有标记 → 检查标记数≥2 且 每步均长≥8字
  → 部门归一化（DeepSeek LLM 批处理）
  → 语义向量匹配（部门过滤 + 本地 numpy 余弦相似度）
  → 三档判定 + 颜色标注
  → 输出 1 个匹配结果 Excel
```

## 2. 输入规范

### 2.1 标准模板（7列，全部必填）

| 列 | 字段 | 说明 |
|----|------|------|
| A | 部门 | 如：推广、运营、财务 |
| B | 需求提交人 | 姓名 |
| C | 工作流程名称 | 简短概括，如「日报填写」 |
| D | 具体工作步骤说明 | **必须分步编号（1. 2. 3.），≥30字** |
| E | 业务痛点 | 如：高频重复、耗时长 |
| F | 涉及到的软件系统和网页 | 如：京东（京准通） |
| G | 企业名称 | 如：北京XX科技 |

### 2.2 步骤详细度要求
| 检查项 | 规则 | 不通过行为 |
|--------|------|-----------|
| 分步标记 | 至少 2 个（`1.` `1、` `①` `一、` `第一步` 等） | 🔴 退回 |
| 每步均长 | ≥ 8 字（去掉编号标记后） | 🔴 退回 |
| 兜底 | 无任何标记时，总字符数 ≥ 30 字 | 🔴 退回 |

> **实际判定逻辑**：优先检测分步标记 → 有标记则按每步均值判定（更宽松）；无标记才用 30 字绝对下限。
>
> **步骤编号正则**（`STEP_MARKER_PATTERN`，已预编译）：`(?:\d{1,2}[\.\、\,\，]|[一二三四五六七八九十]{1,2}[\.\、\,\，]|[①②③④⑤⑥⑦⑧⑨⑩]|步骤\s*\d|第[一二三四五六七八九十\d]{1,2}步)`

## 3. 智能表头映射

### 3.1 触发条件
上传的 Excel 表头与标准 7 列不完全一致时触发，**不会直接拒绝**，而是尝试自动映射。

### 3.2 映射规则
- **精确匹配**：列名与标准列名完全一致时直接映射。
- **LLM 智能映射**：调用 DeepSeek API，传入用户列名与标准列名，LLM 返回对应映射关系。
- **双向包含匹配**：若 LLM 返回值与标准列名存在双向子串包含（如「流程名」↔「流程名称」），则采纳。
- **未映射列**：对应单元格留空，黄色背景 + 批注「⚠ 请补充」。

### 3.3 交付物
生成 `{原文件名}_待补充.xlsx`：
- 已映射列：自动填入原数据，单元格添加批注说明来源列名。
- 未映射列：对应单元格留空，以 **黄色背景** 高亮并标注「⚠ 请补充」。
- 该文件直接交付用户，用户补全后重新上传方可执行语义匹配。

### 3.4 限制
- 严重缺列（如只提供 3 列及以下）可能无法有效映射，此时应直接提示用户按标准模板重新填写。
- 智能映射**不替代语义匹配**，只是前置的标准化步骤；补全后的文件必须再次通过完整校验。

## 4. 匹配引擎

### 4.1 技术栈
| 组件 | 选型 | 说明 |
|------|------|------|
| 向量模型 | `sentence-transformers/all-MiniLM-L6-v2` | 384 维，本地 ONNX，免费 |
| 向量存储 | Supabase PostgreSQL `scenes.embedding` | 仅存储，不在云端做向量检索 |
| 业务存储 | Supabase PostgreSQL `scenes` 表 | 统一存储场景元数据 + 向量 |
| 数据访问 | **Supabase REST API** | `POST/GET/PATCH` + `service_role` key |
| 相似度计算 | **本地 numpy 批量余弦相似度** | `np.stack + np.dot` 向量化计算 |
| 部门归一化 | DeepSeek API 批处理 | `normalize_depts_batch()` 一次性归一化所有部门 |
| Excel 处理 | `openpyxl` | 读写、样式、批注 |
| 依赖 | openpyxl + sentence-transformers + numpy + requests + python-dotenv + huggingface_hub | |

### 4.2 匹配逻辑
- 查询文本 = `工作流程名称` + `\n` + `具体工作步骤说明`
- 本地生成 384 维向量（归一化）
- **部门过滤**：输入行的部门先经 DeepSeek 归一化（如「财务部」→「财务」），只与同部门场景做匹配
- **本地批量匹配**：拉取部门内全部场景的 embedding 列表，用 numpy 计算 `cosine_sim = np.dot(query, scenes.T)`，取最大值
- 只取 **Top-1** 结果

### 4.3 三档判定
| 相似度 | 判定 | 颜色 | 操作 |
|--------|------|------|------|
| ≥ 0.85 | 已有场景 | 🔵 蓝 | 不重复入库 |
| 0.75 ~ 0.85 | ⚠ 待确认 | 🟡 黄 | 人工判断 |
| < 0.75 | 新场景 | 🟢 绿 | 建议入库 |
| 步骤不详细 | 🔴 退回 | 🔴 红 | 补充重新提交 |

## 5. 输出规范

### 5.1 输出列
原始7列 + 7列扩展：

| 扩展列 | 内容 |
|--------|------|
| 流程是否详细 | ✅ 详细 / 🔴 不详细 |
| Top1场景 | 匹配到的最相似场景名称 |
| Top1场景流程 | 匹配场景的步骤说明 |
| 相似度 | 0~1 的余弦相似度 |
| 自动判定 | 已有场景 / ⚠ 待确认 / 新场景 / ⚠ 流程不详细 |
| 建议操作 | 不重复入库 / 人工确认 / 建议入库 / 退回原因 |
| 是否入库 | 留空，供用户回填「是/否」后执行反馈入库 |

### 5.2 输出规则
- 每次只产出 **1 个文件**：`{输入文件名}_匹配结果.xlsx`
- 直接覆盖同名文件，不带版本号
- 不保留任何中间文件
- 若触发智能映射，额外产出 **1 个待补充文件**：`{输入文件名}_待补充.xlsx`

## 6. 数据存储

### 6.1 Supabase PostgreSQL `scenes` 表
```sql
CREATE TABLE scenes (
    id            BIGSERIAL PRIMARY KEY,
    scene_id      TEXT UNIQUE NOT NULL,
    scene_title   TEXT,
    process_steps TEXT,
    department    TEXT,
    owner         TEXT,
    platform      TEXT,
    source        TEXT,
    pain          TEXT,
    embedding     VECTOR(384)
);
```

- `scene_id`：8 位 UUID 哈希（`uuid.uuid4().hex[:8]`），脚本自动生成
- `embedding`：384 维向量，由 `all-MiniLM-L6-v2` 生成
- 历史索引 `ivfflat (embedding vector_cosine_ops)` 仍保留以备未来切换到云端匹配

### 6.2 数据访问通道
- **写入**：`POST /rest/v1/scenes`（插入新场景）或 `PATCH /rest/v1/scenes?scene_id=eq.xxx`（更新）
- **读取**：`GET /rest/v1/scenes?select=...&department=eq.xxx`（按部门过滤拉取）
- **Header**：`apikey: <SUPABASE_SERVICE_KEY>` + `Authorization: Bearer <SUPABASE_SERVICE_KEY>`
- **DB 直连废弃**：`SUPABASE_DB_URL` 已不再使用，保留仅作兼容性占位

## 7. 命令行用法

```bash
python semantic_match.py --input <输入文件.xlsx>
```

| 参数 | 说明 |
|------|------|
| `--input` | 输入 Excel 路径（必填） |
| `--apply-feedback` | 从匹配结果文件中读取「是否入库」列，执行入库或跳过 |
| 输出路径 | 默认 `{输入文件名}_匹配结果.xlsx`，与输入文件同目录 |

阈值固定为 `0.85`（高）/ `0.75`（低），如需调整修改 `semantic_match.py` 中的常量。
